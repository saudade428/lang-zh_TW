**彩色狀態指示器**彩色指示器在*陣列設備*中每行開頭的意義如下：

<i class='fa fa-circle orb green-orb'></i>正常運行，設備處於活動狀態。

<i class='fa fa-circle orb grey-orb'></i>設備處於待機模式（休眠）。

<i class='fa fa-warning orb yellow-orb'></i>設備內容已模擬。

<i class='fa fa-times orb red-orb'></i>設備已禁用，內容已模擬。

<i class='fa fa-square orb blue-orb'></i>新設備。

<i class='fa fa-square orb grey-orb'></i>設備不存在，位置為空。

**標識**是用於唯一識別儲存設備的*簽名*。簽名
包括設備型號、序號、Linux 設備 ID 和設備大小。

**溫度**直接從設備讀取。您可以在 [顯示首選項](/Settings/DisplaySettings)
頁面配置要使用的單位。我們不讀取休眠硬碟驅動器的溫度，
因為這通常會使它們被喚醒，所以改為顯示 `*` 符號。我們還會顯示 `*` SSD 和快閃記憶體設備的符號，
儘管有時這些設備確實報告了有效的溫度，
但有時它們還是返回值 `0`。

**大小, 已用, 可用**報告總設備大小、已用空間和剩餘空間。這些
還可以在 [顯示首選項](/Settings/DisplaySettings) 頁面上配置單位。
由於文件系統開銷，空磁碟使用的空間量也不會為零。

*注意：對於多設備快取池，此數據是 btrfs 返回的整個池的數據。*

**讀取, 寫入**是發送到設備 I/O 驅動程序的 I/O 請求的計數。這些統計數據可以
隨時清除，請參閱下面的陣列狀態部分。

**錯誤**用來對設備 I/O 驅動程序報告的*不可恢復*的錯誤計數。
由於出現不可恢復的陣列讀取錯誤而導致的數據遺失，將使用奇偶校驗重建功能即時填充（並且我們嘗試將此數據寫回發生故障的扇區）。
任何不可恢復的寫入錯誤都會導致*禁用*磁碟。

**FS**指的是在設備的分區 1 中檢測到的文件系統。

**查看**列中有一個文件夾圖示，表示設備已*掛載*。單擊此圖示
瀏覽文件系統。

如果在 [顯示首選項](/Settings/DisplaySettings) 頁面開啟了 "顯示陣列總計"，則
會包含一行**總計**，其中提供設備統計數據，包括設備的
平均溫度。

必須停止陣列才能更改陣列設備分配。

UnRAID 陣列由一個或兩個奇偶校驗磁碟和多個數據磁碟組成。數據
磁碟專門用於儲存用戶數據，奇偶校驗磁碟提供必要的冗餘
以從磁碟故障中恢復。

由於數據不跨陣列條帶化，因此奇偶校驗磁碟必須與最大數據磁碟相同或比它更大。
奇偶校驗磁碟也應當是您性能最高的驅動器。

每個數據磁碟都有自己的文件系統，並且可以
作為獨立共享導出。

單擊設備名稱以配置各個設備設定，並啟動某些實用程序。
:end

**彩色狀態指示器**彩色指示器在*池設備*中每行開頭的意義如下：

<i class='fa fa-circle orb green-orb'></i>正常運行，設備處於可用狀態。

<i class='fa fa-circle orb grey-orb'></i>設備處於待機模式（休眠）。

<i class='fa fa-square orb blue-orb'></i>新設備。

<i class='fa fa-square orb grey-orb'></i>沒有設備，位置為空。

**池設備**是單一設備，或多個設備的池，位於 Unraid 陣列的*外部*。可以像陣列設備一樣將其導出以進行網路訪問。在 Unraid 陣列之外可以顯著加快寫入訪問速度。

有兩種配置池設備的方法：

1. 作為單一設備，或
2. 作為多設備池。

當配置為單一設備時，您可以使用任何受支援的文件系統（btrfs、reiserfs、
或 xfs）格式化該設備。此配置可提供最高的性能，但代價是不提供數據保護 -
如果單一池設備發生故障，其中包含的所有數據都可能會遺失。

當配置為多設備池時，Unraid OS 將自動選擇 *btrfs-raid1* 格式（同時用於數據
和元數據）。btrfs 允許將任意數量的設備添加到池中，並且保證
每個數據副本都可以寫入兩個不同的設備。因此該池可以承受單一磁碟故障，不會遺失數據。

啟用[用戶共享](/Settings/ShareSettings) 時，可以將用戶共享配置為
自動使用池設備，以加快寫入速度。
可以計劃一個稱為 *mover* 的特殊後台進程
定期運行，以將用戶共享文件從快取移到陣列。
:end

:array_devices_help:
#### 分配設備

一個 unRAID 磁碟陣列由一個奇偶校驗磁碟和多個數據磁碟組成。數據磁碟
專門用於儲存用戶數據，而奇偶校驗磁碟提供必要的冗餘，
以從任何單一磁碟故障中恢復。

請注意，在引用陣列儲存設備時，我們會慎用*磁碟*一詞。
當引用實際硬碟驅動器 (HDD) 設備時，我們使用*硬碟驅動器*（有時也稱為*驅動器*）一詞。
這是因為在 RAID 系統中，可以讀取/寫入的陣列磁碟的相應
硬碟驅動器可能已禁用甚至已遺失! 另外，懂得詢問 “將哪個設備
分配為奇偶校驗磁碟？” 或 “哪個設備對應於 disk2？”都很有用。

因此我們需要一種方法來將硬碟驅動器分配給陣列磁碟。當陣列停止時，
可以在首頁上完成此操作。每個陣列磁碟都有一個下拉框，其中列出了所有
未分配設備。要分配設備，只需從列表中選擇它即可。每次進行設備
分配時，系統都會更新設定檔以記錄分配。

#### 要求

與在所有陣列設備上對數據進行條帶化的傳統 RAID 系統不同，unRAID 伺服器
會將文件儲存在各個硬碟驅動器上。因此，所有文件寫入操作都將涉及
所寫入的數據磁碟和奇偶校驗磁碟。鑑於這些原因，

*奇偶校驗磁碟的大小必須大於或等於任何數據磁碟的大小，

以及

*在可選擇的情況下，奇偶校驗磁碟應當是集合中速度最快的磁碟。

#### 指導方針

在設計 unRAID 磁碟陣列時，應遵循以下步驟：

1. 確定要用於奇偶校驗的硬碟驅動器，以及要用於
數據的硬碟驅動器（disk1 、disk2 等），並以某種方式進行標記。另外，找到每個硬碟驅動器的序號並將其記在某處；日後需要使用此資訊。

2. 安裝硬碟驅動器設備，啟動 unRAID 伺服器和 webGui。如果這是全新的
系統構建，首頁將顯示未安裝任何磁碟。這並不意味著系統無法檢測
到硬碟；這只是意味著尚未分配任何磁碟。

3. 還記得在步驟 1 中記錄的序號嗎？對於奇偶校驗磁碟和每個數據磁碟，
請根據其序號從下拉選單中選擇合適的硬碟驅動器。

#### 熱插拔

如果硬體支持，也可以將硬碟驅動器*熱插拔*到伺服器中。例如，
如果使用的是硬碟驅動器架，可以在通電且陣列已停止的情況下
將它們簡單地插入伺服器。刷新首頁後，未分配的新設備將出現在分配
下拉選單中。

#### 後續步驟

分配所有硬碟驅動器後，請參考下面的陣列狀態部分
並啟動陣列。
:end

:encryption_help:
#### 加密輸入

陣列停止後，用戶可以指定一個新的加密金鑰。請注意，當使用
特定金鑰格式化設備後，只有使用相同的金鑰才能將其打開。更改加密金鑰需要
重新格式化已加密設備，**導致這些設備上所有的現有數據永久遺失。**

#### 密碼

輸入最多 512 個字元的密碼。強烈建議僅使用 [ASCII 表](https://en.wikipedia.org/wiki/ASCII) 中前面 128 個字元中的 95 個可列印字元，因為它們始終具有相同的二進製表示形式。
根據系統配置，其他字元可能具有不同的編碼，您的密碼
將無法使用其他編碼。如果您希望使用更長的密碼或包含二進制數據，請改為上傳金鑰文件。

請參考 [cryptsetup FAQ](https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects)
以了解構成*安全*密碼的元素。

**記住**這個密碼。**如果遺失密碼，已加密的內容將無法恢復！**

#### 金鑰文件

選擇具有儲存的加密金鑰的本地金鑰文件或二進制文件。金鑰文件的最大大小為 8M（8388608 位元組）。

**備份**您的本地金鑰文件。**如果遺失，已加密的內容將無法恢復！**
:end

:info_btrfs_balance_help:
**平衡**將運行 *btrfs balance* 程序以重新條帶化所有池設備上的盤區，例如，將池從 raid1 轉換為 raid0，或從 raid0 轉換為 raid1。

當執行*完全平衡*時，基本上是將所有內容重寫到當前文件系統中。

*模式轉換*會影響 btrfs 數據範圍；元數據始終使用 raid1，並在必要時通過任何平衡操作轉換為 raid1。

運行時間可能很長，具體取決於文件系統大小和設備速度。

在創建多設備池時，Unraid 使用以下預設選項：

`-dconvert=raid1 -mconvert=raid1`

有關更完整的文件，請參閱 btrfs-balance [管理](https://btrfs.wiki.kernel.org/index.php/Manpage/btrfs-balance)

*注意：raid5 和 raid6 一般仍被 Linux 社區視為**實驗性***
:end

:smb_case_sensitive_names_help:
控制檔案名是否區分大小寫。

使用默認設置**自動**時，支持區分大小寫檔案名 (Linux CIFSVFS) 的用戶端
可基於封包告知 Samba 伺服器，它們希望以區分大小寫的方式
訪問文件系統（以支持 UNIX 區分大小寫的語義）。Windows 系統不支持區分大小寫的檔案名，因此
將該選項設定為**自動**的效果，與將其設置為“否”相同；但是，會保留 Windows 用戶端傳遞的檔案名的大小寫。對於大型目錄，此設置可能會導致
性能降低，因為 Samba 必須進行檔案名搜索並匹配傳遞的名稱。

設置為**是**表示在用戶端通過的情況下創建文件，並且只能使用相同的大小寫進行訪問。
這將加快大型目錄的訪問速度，但是使用此設置可能會令某些 Windows 應用程式無法正常運行。
例如，如果創建了 'MyFile'，但某個 Windows 應用程式嘗試打開 'MYFILE' （Windows 允許這種情況），會找不到該文件。

**強制小寫**的值很特殊：所有傳入用戶端檔案名（不僅是新檔案名）都將設置為小寫。
換句話說，無論在 Windows 端創建了何種大小寫混用的名稱，都將以全小寫來儲存和訪問該名稱。
這樣可以確保所有 Windows 應用程式正確尋找無論大小寫如何的任何文件，但大小寫不會保留在文件夾列表中。
請注意，僅應為新共享配置此設置。

備註：如果您的路由器或上游 DNS 伺服器啟用了 
[DNS 重新綁定保護](https://en.wikipedia.org/wiki/DNS_rebinding)，**預配**可能會失敗。
DNS 重新綁定保護可防止 DNS 解析到專用 IP 網路範圍。在有漏洞/不安全 "web" 介面的傳統設備所在的本地 LAN 中，DNS 重新綁定保護將作為一項安全特性。

DNS 重新綁定保護的來源之一可能是您的 ISP DNS 服務商。這種情況下可以切換到其他DNS 服務商來解決問題，例如切換到Google的公用 DNS。

而更普遍的情況下，DNS 重新綁定保護可能是在您的路由器中啟用的。大多數消費級路由器不執行 DNS 重新綁定保護；
但如果執行了保護，應當有一條配置設置用於關閉此功能。

然而，較高端的路由器往往會**啟用**DNS 重新綁定保護。通常可以透過一些途徑將其完全關閉，或是根據域名有選擇性地關閉。例如：

**DD-WRT:** 如果您正在使用啟用了 DNS 重新綁定保護的 "dnsmasq"，可以在路由器的設定檔中添加下面這一行：

`rebind-domain-ok=/unraid.net/`

**pfSense:**如果您正在使用 pfSense 內部 DNS 解析器服務，可以添加以下自訂選項行：

`server:`<br>
`private-domain: "unraid.net"`

**Ubiquiti USG router:**您可以添加以下配置行：

`set service dns forwarding options rebind-domain-ok=/unraid.net/`

**OpenDNS:** 轉到“設置 -> 安全”，然後*移除*位於“可疑回應   
 - 阻止內部 IP 地址”旁的複選框。這是一個全有或全無的設置。
 
若以上方法均告失敗，可以在電腦的 *host* 文件中創建一個條目，以覆蓋外部 DNS，
並直接將您伺服器的 unraid.net FQDN 解析到它的本地 IP 地址。
:end

:eth_bonding_mode_help:
**模式 0（平衡 - rr）**<br>
該模式將按順序傳輸封包，從第一個可用的從屬介面一直傳輸到最後一個介面。
如果兩個實際介面是該綁定中的從屬介面，並且有兩個封包到達綁定介面之外的位置，則第一個封包在第一個從屬介面中傳輸，第二個封包在第二個從屬介面中傳輸。
第三個封包在第一個從屬介面中發送，依此類推。
該模式提供負載平衡和容錯。

**模式 1（主動備份） - 默認**<br>
該模式將其中的一個介面置於備份狀態，並且只在主動介面遺失鏈路時啟用該介面。
在某個時間實例內，該綁定中只有一個從屬介面處於活動狀態。只有在活動的從屬介面出現故障時，才會啟用另一個從屬介面。
該模式提供容錯。

**模式 2（平衡 - 異或）**<br>
該模式基於異或公式傳輸封包。源 MAC 地址與目標 MAC 地址通過異或公式對從屬介面數取模。
每個目標 MAC 地址將選擇相同的從屬介面，從而提供負載平衡和容錯能力。

**模式 3（廣播）**<br>
該模式在所有從屬介面中傳輸所有封包。該模式用得最少（僅用於特殊用途），且只提供容錯。

**模式 4（802.3ad）**<br>
該模式被稱作*動態鏈路聚合*。它會創建聚合群組，共享相同的速度和雙工設置。
它需要支持 IEEE 802.3ad 動態鏈路的交換機。
根據傳輸哈希策略對傳出流量的從屬介面進行選擇，可以通過 xmit_hash_policy 選項來從預設的簡單異或策略更改為其他策略。
注意，並非所有的傳輸策略都符合 802.3ad 的要求，尤其是 802.3ad 標準中 43.2.4 章節關於封包錯序的要求。
不同的對等實現對於不合規有著不同的容限。

**模式 5（平衡 - tlb）**<br>
該模式被稱為*自適應傳輸負載均衡*。傳出流量根據每個從屬介面的當前負載和隊列進行分發。
傳入流量由當前從屬介面接收。

**模式 6（平衡 - alb）**<br>
該模式被稱為*自適應負載均衡*。它包括平衡 - tlb 和面向 IPV4 流量的接收負載平衡 (rlb)。
接收負載均衡利用 ARP 協商來實現。
綁定的驅動程序將攔截伺服器發出的 ARP 回應，並且用該綁定中某個從屬介面的唯一硬體地址重寫源硬體地址，
以便不同的用戶端對該伺服器使用不同的硬體地址。

*建議設置為模式 1（主動備份）。其他模式雖然允許您設置特定的環境，但可能需要適當的交換機支持。
選擇不支持的模式可能導致通信中斷。*
:end
